# Base65536 编码原理详解

## 什么是 Base65536

Base65536 是一种基于 Unicode 字符的编码方案，使用 Private Use Area（PUA）中的字符，范围是 U+10000 到 U+1FFFF，共 65536 个不同的字符。

### 编码原理

每个 Base65536 字符可以表示 16 位（2 字节）的数据：

- **原始数据**：二进制字节流
- **编码过程**：每 2 字节 → 1 个 Base65536 字符
- **理论膨胀率**：50%（2 字节 → 1 字符，但 Unicode 字符在 UTF-16/UTF-32 中占用更多空间）

实际使用中，由于 UTF-8 编码：
- ASCII 范围内的字符：1 字节
- 其他 Unicode 字符：3-4 字节（UTF-8）
- 平均膨胀率：约 50-60%

## 与 Base64 的对比

| 特性 | Base64 | Base65536 |
|-----|--------|----------|
| 字符集 | A-Za-z0-9+/ | U+10000-U+1FFFF |
| 每字符位数 | 6 bit | 16 bit |
| 膨胀率 | 133% | ~50% |
| 可读性 | 基本不可读 | 大量特殊字符 |
| 兼容性 | 通用 | 部分系统可能不支持 |

## gzip 压缩

### 何时使用压缩

- **文本文件**：压缩效果显著（70-80%）
- **JSON/XML**：压缩效果显著
- **源代码**：压缩效果良好
- **已压缩文件**：不建议再压缩
- **图片/视频**：压缩效果有限

### 压缩级别

gzip 支持 1-9 级别压缩：

- 1：最快，压缩率最低
- 9：最慢，压缩率最高
- 默认使用级别 9 以获得最佳压缩效果

### 压缩检测

解码时自动检测 gzip 格式：

```python
if data[:2] == b'\x1f\x8b':  # gzip magic number
    data = gzip.decompress(data)
```

## 元数据格式

```json
{
    "original_name": "原始文件名",
    "compressed": true/false,
    "original_size": 12345
}
```

存储在文件首行：`#METADATA:{...}`

### 元数据作用

1. **保存原文件名**：解码时自动还原
2. **标记压缩状态**：解码时决定是否解压
3. **记录原始大小**：用于验证完整性

## 实现注意事项

### 1. 编码边界

Base65536 按 2 字节为单位编码：
- 数据长度为奇数时，最后 1 字节需要特殊处理
- 建议在编码前填充或调整

### 2. Unicode 问题

某些 Unicode 字符可能在特定环境下出现问题：

- 控制字符：应避免使用
- 零宽字符：可能导致文本处理问题
- 代理对：某些语言可能错误处理

### 3. 文件大小限制

- 单个文件建议不超过 100MB
- 大文件可能需要较长的处理时间
- 注意内存占用

## 测试验证

往返一致性测试：

```bash
# 编码
python skill.py encode input.zip -o output.txt

# 解码
python skill.py decode output.txt -o restored.zip

# 验证
sha256sum input.zip restored.zip
# 两个 SHA256 应该完全一致
```

## 常见问题

### Q: 编码后的文本在某些平台显示异常？

A: 这可能是平台的文本渲染问题。Base65536 使用的是合法的 Unicode 字符，如果平台不支持，可能会显示为方框或跳过。建议在支持完整 Unicode 的平台使用。

### Q: 为什么我的文本文件编码后反而变大了？

A: 文本文件本身已经接近最优编码，再压缩可能反而增加体积。对于纯文本，建议禁用压缩（`--no-compress`）。

### Q: 解码失败怎么办？

A: 检查：
1. 文件是否完整（传输过程中可能损坏）
2. 编码时是否使用正确的参数
3. 元数据头是否完整
4. Base65536 库版本是否一致
5. 密钥是否正确
